# 実装計画: 生ドラムのリアルタイムBPM検出 → BOSSルーパー MIDIテンポ同期

> 目的: マイク/オーディオIFから入る**生ドラム**のテンポをリアルタイムに割り出し、**BOSSルーパーへMIDIクロックを送ってテンポ同期**する。

---

## 0. 設計の核心 — ドラマー主導の「純粋追従」と、その安定化

**前提（要件・固定）: ドラマーはクリックを聴かない。** ドラマーが自分のテンポで自由に叩き、**それが唯一のテンポ基準**。ツール側はそれに追従する（マスター=ドラマー、スレーブ=ルーパー）。「ドラマーをクリックに合わせさせて安定化する」方式は要件で却下。

ただし**閉ループは依然として存在する**。ドラマーは自分の録音ループ（＝ルーパー再生音）を聴きながら叩くからだ。

```
ドラマー → (音) → 検出 → 平滑化 → MIDIクロック → ルーパー再生 → (音) → ドラマー …
```

ループ内には合計遅延（入力バッファ + 検出遅延 + 平滑化ラグ + MIDI送出 + ルーパー内部平滑化）がある。**遅延を持つ結合系は、追従ゲインが高いとテンポが発散（走る/もたつくの増幅）し、低ければ安定する。** クリックという外部アンカーが無い以上、安定化の手段は一つしかない:

> **設計原則: ルーパーを「遅くて強く減衰した追従器（低ゲイン）」にする。**
> ドラマーが“狙っているテンポ”のゆっくりした変化だけ追い、1打ごとの数msのゆらぎは**無視**する。

- 低ゲインにする理由: ルーパーが1打ごとのゆらぎに即反応すると、(a) テンポがガタついて音が歪む、(b) それを聴いたドラマーがさらに反応して**発散**しうる。遅く追えば両方を同時に防げる。
- **実質のアンカーはドラマー自身の時間感覚**。外部基準が無いので系全体がゆっくりドリフト（だんだん速く/遅く）し得るが、それは「ループがドラマーに付いていく」という**狙い通りの挙動**であって不具合ではない。避けるべきは“ガタつき”と“発散”だけ。
- よって「内部に滑らかなテンポ状態を持ち、検出値へ**ゆっくり寄せる**（ドリフト補正）」という制御を**出力側の平滑化手法として採用**する。これはドラマーが聴くクリックではなく、あくまでルーパーへ送るテンポを低ゲインで動かすための内部状態。

### cold-start（初期テンポ）をクリック無しで決める

クリックが無いので開始時テンポの獲得方法を決める:
- **(推奨) 自動取得（ACQUIRING）**: ドラマーが叩き始めた最初の数小節で検出器がテンポを獲得 → ロック → 追従開始。ドラマーは何にも合わせない。**最初に録音するループが事実上テンポを定義する**。
- **(保険) 初期値の手入力/タップ**: オペレーターが開始前にタップ等で初期BPMを与える。これは“ドラマーがクリックに合わせる”ことではなく単なる初期値設定。ACQUIRINGが不安定な時用。
- 初期テンポが決まれば検出を「初期±X%」に拘束でき、octave誤り（半分/倍）も大きく抑えられる。

### ハードウェア前提の検証を最優先（プレミス確認）

着手前に実機で確認すべき本質的な未知数：
- **BOSSルーパーは外部MIDIクロックを受けたとき、録音済みループの再生速度を実際に追従（タイムストレッチ）するのか。** 機種により「新規録音/リズム/メトロノームのテンポにしか効かない」場合がある。
- 追従する場合、**テンポ変化時に音が歪む/プチる範囲**はどこまでか（±数%か、±十数%か）。
- ロックに必要な最小クロック数・追従できるテンポ変化幅・更新の粒度。

これが「再生ループも追従する」前提を満たさないと、プロジェクトの目的自体が変わる。**§11 段階1で固定BPMの実機確認を最初にやる理由はこれ。**

## 1. ゴールと要件

- 入力: オーディオインターフェースからの**ライブ生ドラム**（モノ or ステレオ、低レイテンシ）。
- 出力: **MIDIクロック（0xF8, 24 PPQN）** を BOSSルーパーへ送り、ルーパーのテンポを追従させる。
- 必須: 1打ごとの数msのゆらぎを**平滑化**し、ルーパーのテンポがガタつかない（音が歪まない）こと。
- 必須: 系として**安定**であること（§0。追従が発散しない）。
- 形態: PCで動く**スタンドアロンなPythonプロセス**。最初はCLI、後で簡易GUI。

非ゴール（最初は対象外）:
- 拍子・小節頭の高精度推定（まずは4分のテンポ追従に集中）。
- 複雑な譜面解析。
- 位相（小節頭）同期（段階2。まずはテンポの追従に集中）。
- クリック/メトロノームをドラマーに聴かせる方式（要件で却下）。

### 開発・検証方針（生ドラムは最終確認のみ）

生ドラムの実演奏で毎回検証するのは困難なので、**開発・チューニングの主戦場は録音音源（YouTube等）にする**。生ドラムは最後の実機確認だけに使う。

- **「学習」の意味**: 本ツールはDSPベース（aubio/PLL）でMLモデルを学習するわけではない。ここでの学習＝**録音コーパスを使ったパラメータの調整・自動探索（grid/ベイズ探索）**＋**検証用の正解付きコーパス構築**。同じコーパスは、将来DSPで精度不足なら学習型ビートトラッカー（madmom/BeatNet系）の訓練・比較にもそのまま使える（初期スコープ外）。
- **録音で検証できること／できないこと**（重要な切り分け）:
  - ✅ できる: 検出精度・テンポ推定・追従の正確さ・レイテンシ・MIDIクロックの正しさ（=**開ループ**）。
  - ❌ できない: §0の**閉ループ安定性**。録音はルーパー出力に反応しない（フィードバックが無い）ため。→ 閉ループは**シミュレートしたドラマー**（§12）か生ドラムでのみ確認できる。

## 2. 実行環境の重要注意（WSLは不可）

- **WSL2ではオーディオ入力デバイス/MIDIポートに直接アクセスできない**ため、このツールは**Windowsネイティブ（または対象OSのネイティブ）で実行**する。
- 開発も基本ネイティブ環境（Windowsなら PowerShell + venv）。WSLでコードは書けるが、実行・実機テストはネイティブで。
- 仮想MIDIポート: Windows = **loopMIDI**、macOS = **IAC Driver**。動作検証に使う。
- **Windowsのタイマ分解能**: 既定では `time.sleep` の粒度が粗い（~15.6ms）。クロックスレッド開始時に `timeBeginPeriod(1)`（`ctypes` で `winmm`）を呼び 1ms に上げる。終了時 `timeEndPeriod(1)`。Python 3.11+ は内部で改善されているが、明示しておくと確実。

## 2.5 必要なハードウェア構成（実機: BOSS RC-600 / ZOOM UAC-232）

> どちらも**未入手**。購入前に下記の「購入前に確認」を満たすか要チェック（特に RC-600 の外部MIDIクロック追従仕様）。

### 機材リスト

| 区分 | 機材 | 役割 / 備考 |
|---|---|---|
| ルーパー | **BOSS RC-600**（確定） | MIDIクロック受信でテンポ同期。MIDI IN/OUT は**3.5mm TRSジャック**（DIN変換アダプタが付属。要確認）。USBもMIDI/オーディオに使える（クラスコンプライアント想定だが要確認） |
| オーディオIF | **ZOOM UAC-232**（確定予定） | 2in/2out、USB-C、**32bitフロート録音**（入力クリップ無し＝ドラムの強打でもオンセットが潰れない利点）。**MIDI IN/OUT 端子あり**（端子形状＝DIN/3.5mm かは要確認）→ オーディオ入力とMIDIクロック出力を1台で兼ねられる |
| マイク | **キック用ダイナミック1本**（推奨最小構成） | テンポ検出はキックのトランジェントが最も安定。例: AKG D112 / Shure Beta52 等。余裕があればオーバーヘッド1本を input2 に |
| ケーブル | XLR（マイク分）、USB-C（UAC-232↔PC） | |
| MIDIケーブル | UAC-232 MIDI OUT ↔ RC-600 MIDI IN（＋必要なら端子変換） | 経路A用。両者の端子形状（DIN/3.5mm TRS）を確認して適合ケーブル/変換を用意 |
| モニター | ドラマー用ヘッドホン/IEM or モニタースピーカー | **ドラマーが聴くのはクリックではなく自分のループ（ルーパー再生音）**。これが§0の閉ループの実体で、ループ演奏には必須 |
| PC | Windowsネイティブ機 | §2 |
| 電源 | RC-600用ACアダプタ(PSA-S)/電池、PC電源 | |

### MIDIクロックの送出経路（UAC-232 にMIDI I/Oがあるので1台で完結できる）

UAC-232 に MIDI I/O があるため、**オーディオ入力もMIDIクロック出力も UAC-232 一台で賄えるのが最大の利点**。優先順:

- **経路A（推奨）: PC → USB → UAC-232 → MIDI OUT → RC-600 MIDI IN**
  - PCは UAC-232 のMIDI出力ポートへ 0xF8 を送るだけ。**USBデバイスが1つ（UAC-232）に集約**され、オーディオとMIDIのクロック源が同一デバイスになるので構成がシンプル。
  - 要確認: UAC-232 と RC-600 の MIDI端子形状（DIN ↔ 3.5mm TRS）の組み合わせ → 適合ケーブル or 変換（BOSS BMIDI-5-35 等）。
- **経路B（代替）: PC → USB → RC-600（RC-600のUSB-MIDIへ直接）**
  - RC-600 がクラスコンプライアントUSB-MIDIで Windows から出力先に見える場合。ケーブル変換不要。要確認。
- **経路C（最後の手段）: 別途 USB-MIDIインターフェース**
  - A/Bが不可/不安定な場合のみ。追加機材が要る。

> 経路Aを基本線にする。オーディオとMIDIが同一USBデバイスだとタイミングの取り回しも素直。

### BOSS RC-600 側の設定（要マニュアル確認）

- MIDI の **Sync/Clock Source を「外部（EXTERNAL/AUTO）」** に設定し、受信MIDIクロックにテンポ追従させる。
- **MIDI受信チャンネル**、USB と TRS MIDI の**どちらを入力に使うか**を設定で合わせる。
- **録音済みループの再生が外部クロックに追従するか**（テンポ変化で再生速度が変わるか）、その時の**音質（タイムストレッチの歪み/プチノイズ）と追従可能なテンポ幅**を確認（§0 の最重要プレミス）。
- ロック条件（必要クロック数）・Start/Stop の挙動・最小/最大追従テンポを取説で確認。

### ZOOM UAC-232 側の設定

- Windows は **ZOOM専用ASIOドライバ**を入れ、低レイテンシは **ASIO もしくは WASAPI排他**で。`sounddevice`(PortAudio) は WASAPI/WDM-KS が使える。ASIO を使う場合は ASIO対応の PortAudio が必要（使えなければ WASAPI排他で十分低遅延）。
- **入力は32bitフロート**で取得（`dtype='float32'`）。UAC-232はクリップしない設計なのでゲイン追い込み不要 → ドラムの過大入力でオンセット検出が破綻しにくい。
- サンプルレートは 48000 固定運用が無難（aubio設定と一致させる）。blocksize は 128〜512 で実測してglitchが出ない最小に。

### 購入前に確認

- RC-600: **外部MIDIクロックで録音ループ再生が追従するか**（しないなら本プロジェクトの目的が変わる）。MIDI IN の端子形状（3.5mm TRS / DIN）。
- UAC-232: **MIDI OUT 端子の形状**（DIN/3.5mm）と、RC-600 MIDI IN と繋ぐ適合ケーブル/変換（経路A）。Windows用ASIO/低遅延ドライバの提供。入力レイテンシ実測値。**PCからUAC-232のMIDI OUTへ送れるか**（ドライバがMIDIポートを公開するか）。
- マイク: コンデンサなら UAC-232 の **+48Vファンタム**で給電（キック用ダイナミックなら不要）。

## 3. 全体アーキテクチャ

```
[Audio IF] --PortAudio callback(コピーのみ)--> [Ring Buffer]
   --hop単位に再バッファ--> [Beat Detector] --beat events(サンプル時刻)-->
   [Tempo Estimator: PLL/回帰 + 平滑化] --smoothed BPM-->
   [Controller: 状態機械/低ゲイン追従(ドリフト補正)] --target BPM-->
   [MIDI Clock Sender] --0xF8 @24PPQN--> [BOSS Looper]
```

スレッド構成（タイミング独立のため分離）:
- **PortAudioコールバック（RTスレッド）**: ここでは**numpyバッファをコピーしてキュー/リングバッファへ入れるだけ**。重い処理・GC誘発・ログ出力を絶対に置かない（sounddeviceはバッファを再利用するためコピー必須）。
- **解析スレッド**: リングバッファから aubio の hop サイズに**再バッファ**して逐次検出 → ビートイベント生成 → 推定器更新。
- **MIDIクロックスレッド**: `time.perf_counter()` ベースで一定間隔に F8 を送出。BPM変更は共有変数（ロック/atomic）で受け取り、間隔を滑らかに更新。
- **メインスレッド**: 設定・CLI/GUI・状態表示・タップテンポ受付。

## 4. 使用ライブラリ

| 役割 | 第一候補 | 備考 |
|---|---|---|
| 音声入力 | `sounddevice`(PortAudio) | 低レイテンシのコールバック入力 |
| リアルタイム拍検出 | `aubio`(`aubio.tempo`) | **librosaはオフライン向けなので不採用**。aubioは逐次処理向き |
| （高精度代替） | `madmom` のオンラインビートトラッカー | 精度↑だが重く・レイテンシ↑。依存も重い。要検討 |
| （依存回避の代替） | **numpy自前**（スペクトルフラックス+自己相関/櫛フィルタ） | aubio/madmomのビルド地獄を避けたい時。§14参照 |
| MIDI送信 | `mido` + `python-rtmidi` | クロック/Start/Stop送出 |
| 数値処理 | `numpy`（必要なら`scipy`） | バッファ・フィルタ・回帰 |
| 音源取得（検証用） | `yt-dlp` + `ffmpeg` | YouTube等の音声を取得しWAV化（ローカル検証用。§12/§14） |
| オフライン読込 | `soundfile`（+`numpy`） | WAVを读んでファイル入力ソースに流す |
| 参照ビート生成（検証用） | `librosa` or `madmom`（オフライン） | YouTube音源の**グラウンドトゥルース（正解ビート/BPM）**作成。リアルタイム経路には載せない |

> Librosa/Mido について: **Mido は採用**（MIDI送信）。**Librosa はリアルタイムに不向き**なので、検出は aubio（または madmom オンライン）。**Librosaは「録音済みファイルの一括解析＝検証用グラウンドトゥルース作成」用として明確に採用**（リアルタイム経路とは別物）。

> ⚠️ **aubio の依存リスク（重要）**: aubio はリリースが古く、新しめの Python（3.11/3.12）向けの prebuilt wheel が無いことが多く、ビルドに失敗しやすい。対策の優先順:
> 1. **Python 3.10 を pin** して `pip install aubio`（wheel が通りやすい）。
> 2. conda-forge の `aubio` を使う。
> 3. それでも詰まるなら **numpy自前のテンポ検出**（§14）に切替。依存が numpy だけになり長期保守が楽。生ドラム特化なら自前の方が結局調整しやすい場合がある。

## 6. BPM検出（beat_detector）

- `aubio.tempo("default", win_size, hop_size, samplerate)` を使用（例: win=1024, hop=512, sr=48000）。
- 入力ストリームの hop ごとに `tempo(frame)` を呼び、ビート検出を取得。
- **ビート時刻は「累積サンプル数 / sr」で持つ**（壁時計でなく）。
  - 解析スレッドで「これまでに aubio へ渡したサンプル総数」`n_samples` を保持し、ヒット時刻 `t = n_samples / sr`（hop境界精度）。`perf_counter()` で取るとblocksize量子化ジッタ（±blocksize）が混入するので使わない。
- **瞬時BPMは aubio の `get_bpm()` をそのまま使わず**、自前で「ビート時刻」から推定（§7のPLL/回帰）。平滑化と相性が良い。

ドラムのオンセット安定化（**注意: ハイパスは誤り**）:
- キックは**低域（~50–120Hz）**、スネアは**中域+広帯域ノイズ（~150–250Hz）**。ハイパスをかけるとキックの基音が消えるので逆効果。
- 推奨: フルバンドの**スペクトルフラックス**でオンセットを取る（aubioなら onset method を `specflux`/`hfc` で比較）。
- さらに堅牢にするなら**帯域分割**（低域=キック、中域=スネア）でそれぞれエンベロープ/フラックスを取り、キックの周期から4分を推定。生ドラム特化なら効く。

octave error（半分/倍テンポ誤検出）対策:
- 期待BPMレンジを設定（例: 60–200）。範囲外は ×2 / ÷2 して収める。
- 直前の平滑BPMとの連続性チェック（急に倍/半分に飛んだら補正）。
- **初期テンポが決まった後（ACQUIRINGロック or タップ初期値）**は、検出を「現在テンポ ±X%」に拘束でき、octave誤りはほぼ消える。クリック無しなので最初の数小節のACQUIRINGだけは検出が頼り → ここを堅くする。

### 実装で判明した知見（実音源検証）— autocorr + テンポprior を既定に採用

YouTube実音源4曲（183/180/90/120 BPM）で検証した結果:
- **オクターブ誤りが最大の難所**。離散オンセットのIBI方式は密なオンセット（歪んだギター等）で破綻し、**高精度な librosa ですら prior 無しでは4曲中3曲をオクターブ誤り**（183→92, 180→91, 90→181）。これはバグでなくテンポ推定の本質的曖昧性。
- 対策として検出を **「オンセット強度包絡の自己相関 ＋ 対数正規テンポprior」**（`tempogram.py` / `AutocorrTempoDetector`）に変更し**既定**に。
  - 自己相関は**不偏正規化（ac[k]/(N−k)）**でラグ間のピーク高を揃え、オクターブ判断を prior に委ねる。
  - **prior の中心は期待テンポ（`--expected-bpm` / seed / 現在テンポ）**。これを与えると 180/90/120 を正解。
- ブラインド（prior=120既定）では速い曲は半分に折り返す。**運用では概テンポを与える前提**（ドラマー主導でも概テンポは既知/タップ可能）にするのが現実解。
- 残課題: 極端な高速・歪み（Territorial Pissings 183）は prior を与えても外す（onset包絡が飽和）。単一ドラムキットはより素直なはず。

## 7. ★テンポ推定 + 平滑化（tempo_estimator）— 最重要

生ドラムは1打ごとに数ms前後する。**そのままルーパーに送るとテンポが激しくガタつく**。さらに§0の安定性のため、推定は「滑らかで、ゆっくり追従」が条件。

### 7-1. 推定器の選択（v1 → v2）

**v1（シンプル・着手用）: 中央値IBI**
- 直近 N=4〜8 個の IBI（拍間隔）をリングバッファ保持。
- 外れ値除去 → 中央値 → `bpm_raw = 60 / median(IBIs)`。

**v2（推奨・「ガタつかない」要件に直結）: ビート時刻のPLL もしくは最小二乗回帰**
- **最小二乗回帰**: 連続する直近Nビートの時刻 `t_k`（k=0..N-1, 各kは整数拍）に対し `t_k ≈ a + b·k` を最小二乗フィット。傾き `b` が拍周期 → `BPM = 60/b`。全点で傾きを拘束するため単発ジッタに極めて強い。欠落/余剰ビートは `round((t-a)/b)` で拍番号を割り当てて検出・除外。
- **PLL（位相同期ループ）**: 周期 `P` と位相 `φ` を保持。各ビート `t` で予測拍 `φ+P`、誤差 `e = t-(φ+P)`、更新 `φ += P + Kp·e`、`P += Ki·e`（`P`はレンジクランプ）。`Kp,Ki` を小さく取れば本質的に「滑らかな追従器」になり、後段の平滑化を兼ねる。**位相も持つので将来の小節頭ロック（§10）に直結**。

> 採用方針: 着手は v1。安定して end-to-end が回ったら **v2（まず回帰、最終的にPLL）**へ。PLLは推定と平滑化と位相を一体で扱えるので最終形に向く。

### 7-2. 平滑化（推定器の後段／controllerの低ゲイン追従の前）

- **外れ値除去**: 現在の中央値/予測から ±30% を超えるIBIは無視（ミスヒット/ゴースト対策）。
- **スルーレート制限**: 1更新あたりの変化量を **±0.5〜1.0 BPM** にクランプ。
- **デッドバンド（ヒステリシス）**: 変化が **±0.3 BPM 未満**なら送信BPMは据え置き（微振動カット）。ただし**同方向の微小ドリフトが累積するよう、据え置き中も内部目標は更新**（恒常オフセット防止）。
- 一次IIR（指数移動平均）で更にスムーズに: `bpm_out += α*(bpm_target - bpm_out)`（α=0.1〜0.3）。
- **低ゲイン追従＝ドリフト補正（controller側）**: 上記で得た「検出BPM」を直接出さず、内部に持つテンポ状態へ**毎秒 ±数% 以内**でゆっくり寄せてから送出する。クリック無しの本構成では、これが§0の安定性（ガタつき/発散の回避）の肝。

擬似コード（v1 + 平滑化。バグ修正済み）:
```python
class TempoEstimator:
    def __init__(self, n=6, min_bpm=60, max_bpm=200,
                 max_step=1.0, deadband=0.3, alpha=0.2):
        self.ibis = collections.deque(maxlen=n)
        self.bpm_out = None        # 送信用（平滑後）
        self.bpm_target = None     # 内部目標（デッドバンドでも更新）
        self.min_bpm, self.max_bpm = min_bpm, max_bpm
        self.max_step, self.deadband, self.alpha = max_step, deadband, alpha
        self.last_beat_t = None

    def on_beat(self, t):  # t: ビート時刻(秒, サンプル基準)
        if self.last_beat_t is None:
            self.last_beat_t = t
            return self.bpm_out
        ibi = t - self.last_beat_t
        self.last_beat_t = t
        if ibi <= 0:
            return self.bpm_out
        bpm = 60.0 / ibi
        # octave補正（ibiも整合させる）
        while bpm < self.min_bpm: bpm *= 2; ibi /= 2
        while bpm > self.max_bpm: bpm /= 2; ibi *= 2
        # 外れ値除去（中央値比）
        if self.ibis:
            med = statistics.median(self.ibis)
            if abs(ibi - med) > 0.30 * med:
                return self.bpm_out      # 無視（現状維持）
        self.ibis.append(ibi)
        target = 60.0 / statistics.median(self.ibis)
        self.bpm_target = target          # 内部目標は常に更新
        if self.bpm_out is None:
            self.bpm_out = target
            return self.bpm_out
        # スルーレート制限 → IIR → デッドバンド
        step = max(-self.max_step, min(self.max_step, target - self.bpm_out))
        cand = self.bpm_out + self.alpha * step
        if abs(cand - self.bpm_out) >= self.deadband:
            self.bpm_out = cand
        return self.bpm_out
```

パラメータ（N, max_step, deadband, α, PLLの Kp/Ki）は実機で要チューニング。**応答性 ↔ 安定性のトレードオフ**を§12の客観指標（整定時間・追従誤差・BPM標準偏差）で詰めてから耳で最終調整。

## 8. MIDIクロック送出（midi_clock）

- BOSSルーパーは **MIDIクロック(0xF8) を 1拍=24パルス** で受けてテンポ同期。
- 送出間隔: `interval = 60.0 / (bpm * 24)` 秒。BPMが変わったら interval を更新（フィルタ済みなので滑らか）。
- **重要なのは per-pulse のジッタより「累積ドリフトが無いこと」**。受信側（ルーパー）は多数パルスを平均するので 1パルスの数msズレは吸収される。逆に `next_t += interval` で**理想時刻を累積**しないと長時間で系統的にずれる。
- タイミング: **専用スレッド + `time.perf_counter()`**。`time.sleep` は粗いので「sleepで粗く待ち→最後はスピンで合わせる」ハイブリッド。
  - Windowsは§2の `timeBeginPeriod(1)` を併用。
  - スピンは1コアを食う（48msg/sで最後1msスピン ≈ 数%）。許容範囲だが**クロックスレッドにGC誘発処理を置かない・優先度を上げる**（Windowsは `ctypes` で thread priority、可能なら）。
- 必要に応じて:
  - ループ開始時に **Start(0xFA)**、停止時に **Stop(0xFC)**、復帰に **Continue(0xFB)**。
  - **無音/演奏停止では Stop を送らない**（ルーパーが止まる）。クロックは「最後のテンポを維持して流し続ける」（§9-controller）。
  - 位相合わせ: 検出した拍頭でStartを送ると小節頭が揃いやすい（段階2）。

擬似コード:
```python
class MidiClock(threading.Thread):
    def __init__(self, port, bpm=120.0):
        super().__init__(daemon=True)
        self.port = port
        self._bpm = bpm
        self._running = True
    def set_bpm(self, bpm):            # 解析/controllerスレッドから随時更新
        self._bpm = max(20.0, bpm)
    def run(self):
        # Windows: winmm.timeBeginPeriod(1) をここで
        next_t = time.perf_counter()
        while self._running:
            self.port.send(mido.Message('clock'))   # 0xF8
            interval = 60.0 / (self._bpm * 24.0)
            next_t += interval                       # 理想時刻を累積（ドリフト無し）
            delay = next_t - time.perf_counter()
            if delay > 0.002:
                time.sleep(delay - 0.001)            # 粗待ち
            while time.perf_counter() < next_t:      # 残りスピン
                pass
        # Windows: winmm.timeEndPeriod(1)
```

## 9. コントローラ（状態機械）と低ゲイン追従

クリック無し＝ドラマー主導なので、開始は**ドラマーが叩き始めること**で起こる。検出が途切れても破綻しないよう明確な状態遷移を持つ。

```
IDLE ──(入力検知・自動)──────> ACQUIRING
IDLE ──(任意: タップ/初期BPM)─> LOCKED(クロック送出開始)   # ACQUIRINGの保険
ACQUIRING ──(Nビート安定)────> LOCKED(クロック送出開始)    # 通常はこの経路
LOCKED ──(無音 > T秒)────────> HOLD(最後のBPMで送出継続, Stopは送らない)
HOLD ──(演奏再開)───────────> LOCKED
任意 ──(octave/連続性異常多発)─> 補正 or HOLD
```

- **ACQUIRING（cold-start）**: クリックが無いので、開始テンポは**ドラマーの最初の数小節から検出器が獲得**する。安定したら `LOCKED`。ここだけは検出が全てなので堅牢に（§6のレンジ拘束・octave対策）。
- **低ゲイン追従**: `LOCKED` 中は `controller` が「検出BPM」を直接 `midi_clock.set_bpm` に流さず、**内部テンポ状態を毎周期ごく小さく検出値へ寄せる**（例: `tempo += clamp(detect - tempo, ±0.05 BPM/更新)`）。これでルーパーが遅い追従器になり、ガタつき・発散を防ぐ（§0）。
- **無音検出**: 入力レベル（RMS）が一定時間しきい値以下 → `HOLD`。クロックは止めない（Stopを送るとルーパーが停止する）。
- **再ロック**: 演奏再開で検出が戻ったら `LOCKED` へ。

## 10. レイテンシ・位相合わせ

### レイテンシ収支（概算・要実測）

| 要素 | 目安 | 備考 |
|---|---|---|
| 入力バッファ | blocksize/sr（256/48k≈5.3ms, 512≈10.7ms） | 小さいほど低遅延だがCPU↑/glitch↑ |
| 検出遅延（aubio） | 数 hop〜数十ms | 周期確定にオンセット数個必要 |
| 推定/平滑化ラグ | 設計値（**意図的に遅らせる**） | 安定性のため。ここが支配的 |
| MIDI送出+USB | ~1–3ms | python-rtmidi/OSスタック |
| ルーパー内部平滑 | 不明（要実測） | 機種依存。数百msのことも |

→ **テンポ追従なら合計遅延は「数拍遅れて反映」程度で実用上問題ない**。位相（小節頭）合わせには効くので、位相は段階2に分離。

- 最初は「テンポ（速さ）だけ」合わせ、**位相（小節頭）合わせは段階2**（PLLの位相 or 拍頭Startで実現）。
- **タップテンポ/手動ナッジを一級機能**として用意（ACQUIRINGの保険＝任意の初期値設定。検出が乱れた時のフォールバックでもある）。ドラマーがこれに合わせて叩くわけではない（クリック禁止）。

## 12. テスト/検証（録音音源中心・客観指標）

「耳で」だけでなく**数値で**詰める。生ドラムは最終確認のみ（§1方針）。検証は**3つの入力経路**を使い分ける。

### 12-1. 入力経路（3種）

| 経路 | 用途 | 何を検証 | MIDI/レイテンシ |
|---|---|---|---|
| **バッチ（オフライン）** | `eval/batch_eval.py` がWAVを高速処理 | 検出精度・推定・追従の**正確さ**を大量・再現可能に | 載らない（純アルゴリズム） |
| **実時間ファイル再生** | `sources/file.py` を実時間で流す | 上記＋MIDIクロック出力・整定の時間挙動 | 載る |
| **ループバック** | `sources/loopback.py` でブラウザのYouTube再生音をそのまま入力 | **本番に最も近い全経路**（入力→検出→MIDI） | 載る |

> ループバックの作り方（Win）: VB-CABLE/VoiceMeeter or WASAPI loopback（sounddevice）、または UAC-232 のループバック機能。これで**ダウンロードせずに**YouTube再生音を入力できる（ToS的にも無難）。

### 12-2. YouTube検証コーパスとグラウンドトゥルース

- **段階的コーパス**（`eval/corpus.json` にURL/既知BPM/tier/ライセンスを記録。**音声ファイルはコピーライト/容量のためコミットしない**）:
  - **tier1: ドラム単体/ドラム主体**（ドラムループ、練習用ドラムトラック）＝本番（生ドラム単体）に最も近い。**主役**。
  - **tier2: フルミックス＋強い定常ビート**（打ち込み系/クリック制作のポップ）＝ロバスト性確認。BPM一定なので正解が簡単。
  - **tier3: テンポが揺れる/ライブ/ルバート**＝追従・安定性のストレステスト。
  - メトロノーム/クリック動画＝正解が自明な最小確認。
- **グラウンドトゥルース（正解）の作り方**:
  - 定常BPM曲: 既知BPM値（曲データベース or 手タップ）1つで足りる。
  - 揺れる曲: `eval/reference.py` が **madmom/librosa（オフライン・高精度）でビート時刻を注釈** → 軽量リアルタイム検出器の参照にする（強い解析を“先生”にする）。必要なら手タップで補正・キャッシュ。
- ⚠️ **転移の注意**: フルミックス（tier2/3）で詰めた係数が**ドラム単体（本番）に最適とは限らない**。tier1を一次評価に据え、tier2/3はロバスト性確認に使う。

### 12-3. 指標（開ループ＝録音で測れる）

`metrics.py` がログ。**生の検出精度と、意図的に遅らせた追従出力は分けて評価**する。

- **検出精度（瞬時）**: 検出ビート時刻 vs 正解ビートのF値/誤差、瞬時BPM誤差、**octave誤り率**。
- **追従出力**（低ゲイン追従後・遅れて当然）:
  - **整定時間**: 真値ステップ（例 120→126）で出力が ±0.5BPM に収まる秒数。
  - **ランプ追従**: rit./accel.（連続変化）への追従遅れ。
  - **定常ジッタ**: 安定入力での出力BPM標準偏差（「ガタつき」の定量＝最小化対象）。
  - **ACQUIRING成功率/ロック時間**: 冒頭からロックまで（クリック無しなので重要）。

### 12-4. 閉ループ安定性（録音では測れない → 模擬ドラマー）

録音はルーパー出力に反応しないので§0の安定性は測れない。`eval/sim_drummer.py` で**反応するドラマー**を模擬:

- ドラマーを「自分のテンポ意図 + 位相ノイズ」で拍を生成し、**聴こえたルーパークロックへゲイン g・遅延 τ で部分的に引き込まれる**振動子としてモデル化。
- この模擬ドラマー → 検出 → 追従 → クロック → 模擬ドラマー、で閉ループを回し、**発振せず収束するか**を確認。
- **g・τ・自前の追従ゲインをスイープして安定限界（発振する閾値）を測る** → 実機の追従ゲイン上限の根拠にする。

### 12-5. パラメータ自動チューニング（＝「学習」）

- `eval/param_sweep.py` が tier1 コーパスに対し N/max_step/deadband/α/Kp/Ki を **grid または ベイズ最適化**で探索。
- 目的関数: 「定常ジッタ最小」かつ「整定時間が閾値以内」かつ「octave誤り率最小」の多目的（重み付け or パレート）。閉ループ安定余裕（12-4）を制約に入れる。
- 得た最良セットを `config` の既定値に。最後に耳と実機で微調整。

### 12-6. 実機確認（最後だけ）

- 仮想MIDIポート + MIDIモニタ（MIDI-OX(Win) / MIDI Monitor(mac)）でクロックを目視。
- ルーパー実機（RC-600）でロック・歪みの有無を耳と録音で最終確認。
- 生ドラムは最終段でのみ（レイテンシ実測・閉ループの実地確認）。

## 13. リスク・注意

- **閉ループの発散（最重要）** → クリック無し＝ドラマー主導なので外部アンカーは使えない。**ルーパーを低ゲイン・強減衰の追従器にする**（ドリフト補正）。アンカーはドラマー自身の時間感覚。追従ゲインを上げ過ぎない。
- **ルーパーが再生ループを追従しない/歪む** → §0・段階1で実機検証。premiseが崩れたら設計変更（新規録音テンポのみ合わせる等）。
- **テンポ検出のoctave誤り**（半分/倍）→ レンジ制限＋連続性補正＋**ロック後の現テンポ±X%拘束**。クリック無しなのでACQUIRING（最初の数小節）が一番危険 → ここを堅く。
- **Pythonのタイミングジッタ** → 専用スレッド + perf_counter + 累積時刻 + スピン + Windowsは`timeBeginPeriod(1)`。重い処理を同スレッドに置かない。
- **aubioのインストール**が新しめPythonで困難 → Python3.10 pin / conda-forge / **numpy自前**（§14）へ。
- **平滑化の効かせ過ぎ**は追従が遅れる → §12の整定時間で定量し、実演奏で最終調整。
- **コールバックでの重い処理/非コピー参照** → glitch・データ破壊の原因。コールバックはコピー＋enqueueのみ。
- **検証音源の転移リスク** → YouTubeのフルミックスで詰めた係数が本番（ドラム単体）に最適とは限らない。tier1（ドラム主体）を一次評価に（§12-2）。
- **録音では閉ループを検証できない** → §0の安定性は模擬ドラマー（§12-4）か生ドラムでのみ確認可能。録音検証＝開ループまで、と割り切る。
- **グラウンドトゥルースの質** → 揺れる曲の正解はmadmom/librosaの自動注釈が誤ることもある。tier1や定常BPM曲を主軸にし、必要に応じ手タップ補正。
- **YouTube音源の取り扱い** → ダウンロードはToSグレー。**ローカル検証限定**、可能なら**ループバック取込（ダウンロード不要）**やCreative Commons/ロイヤリティフリー素材を優先。音声はリポジトリにコミットしない（URL+BPMのマニフェストのみ）。

## 14. 依存・セットアップ（ネイティブ環境）

```powershell
# Windows (PowerShell)。aubio wheel の都合で Python 3.10 を推奨
py -3.10 -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install sounddevice mido python-rtmidi numpy
pip install aubio            # 失敗時: conda-forge or §14末の自前検出へ
pip install soundfile        # ファイル入力ソース用
# 検証用（YouTube取得・参照ビート生成）
pip install yt-dlp librosa   # ffmpeg も必要（後述）
# madmom を使う場合は別途（依存が重い・古いnumpy要求あり）
```
```bash
# mac
python3.10 -m venv .venv && source .venv/bin/activate
pip install sounddevice aubio mido python-rtmidi numpy
```

- Windows: 仮想MIDIに **loopMIDI**、確認に **MIDI-OX**。
- mac: **IAC Driver**（Audio MIDI設定）。
- **ffmpeg**: yt-dlp/librosa のデコードに必須。Windowsは `winget install Gyan.FFmpeg` 等でPATHに通す。
- **オーディオ・ループバック（YouTube実時間検証用）**: Windows = **VB-CABLE** か **VoiceMeeter**（再生をソフト入力へ回す）、または sounddevice の **WASAPI loopback**（出力デバイスを録音）。UAC-232 本体のループバック機能でも可。これで**ダウンロードせず**にブラウザ再生音を入力できる。

**aubioが入らない/不安定なときの自前検出（numpyのみ）**:
- hopごとにスペクトル（rfft）→ 帯域別エネルギー → **スペクトルフラックス**でオンセット関数。
- オンセット関数の直近数秒に**自己相関 or 櫛フィルタ**をかけてテンポ周期を推定、BPMレンジで拘束。
- 検出した拍時刻を §7 のPLL/回帰へ。依存が numpy だけになり保守が楽で、生ドラム特化のチューニングがしやすい。
